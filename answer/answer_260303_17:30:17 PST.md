## Notes

- **Azure AI Foundry Hub** requires `Contributor` at the Resource Group level to create the Hub and its auto-provisioned linked resources (Storage, Key Vault, ACR, App Insights). The roles in Column 4 are assigned to the **Hub's managed identity** on each connected resource.
- **Azure AI Foundry Projects** can be created with `Azure AI Developer` on the Hub — a less-privileged delegation model is available once the Hub is in place.
- **Azure OpenAI** has no management-plane dependency on other resources, but when used with **On Your Data (RAG)**, its managed identity needs `Search Index Data Reader` on Azure AI Search.
- **Azure Machine Learning** workspace creation requires `Contributor` at the Resource Group — this auto-creates linked Storage, Key Vault, ACR, and App Insights. The roles in Column 4 are for the **AML workspace managed identity**.
- **Azure AI Services** (multi-service Cognitive Services account) has no required cross-resource dependencies at creation time. Runtime integrations (e.g., with AI Search for integrated vectorization) are configured per use case.
- **Azure AI Search** indexer managed identity needs `Storage Blob Data Reader` on the source storage account to pull documents in pull-mode indexing.
- **Azure Bot Service** has no purpose-built management-plane role — `Contributor` on the Resource Group is the minimum. The bot's Entra ID App Registration requires **Application Administrator** (an Entra ID role, not Azure RBAC).
- **Azure AI Document Intelligence** custom model training requires a storage account with labeled training documents — the resource's managed identity or service principal needs `Storage Blob Data Reader` on that container.
- **Application identities** (managed identities for apps calling these services) should use narrower data-plane roles: `Cognitive Services OpenAI User` for OpenAI inference, `Search Index Data Reader` for AI Search queries, `Cognitive Services User` for AI Services/Document Intelligence inference.
