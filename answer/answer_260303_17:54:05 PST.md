The resource file itself covers **both** contexts. From `azure-sql-database.md`:

> **Landing Zone Context**: `Workload Landing Zone / Data Landing Zone`

The placement in `workload-landing-zone/` reflects its **primary** ownership pattern:

- **Workload Landing Zone** — Azure SQL Database most commonly serves as the transactional (OLTP) relational store for an application team. The app team owns the database, manages schema, and controls connections from their app tier. This is the dominant use case.

- **Data Landing Zone** — SQL Database can also appear as a data mart or staging layer (quoted directly from the Overview: *"In a Data Landing Zone it may serve as a data mart or staging layer"*). However, this is a secondary role — data-native workloads in this repo's data landing zone are covered by purpose-built analytics resources: `azure-synapse-analytics`, `azure-data-lake-storage-gen2`, `azure-databricks`, `azure-data-explorer`, etc.

The distinction follows the Cloud Adoption Framework ownership model:
- If an **app/dev team** owns the database → Workload Landing Zone
- If a **data engineering team** uses it as a processing sink → it would logically be a dependency of a data-tier resource (e.g., ADF, Synapse), not a standalone data-zone resource

The file was placed in `workload-landing-zone/` because that's where the RBAC ownership and provisioning responsibility most commonly sits. Its dual-context metadata acknowledges the secondary data use case.
